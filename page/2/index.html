Elastic Search 시스템 환경 구조

elasticsearch는 Lucene을 베이스로 개발한 분산 검색엔진 이다. 설치와 서버 확장이 편리하다. 분산 시스템이기 때문에 검색 DataSet 용량이 증가했을 때 처리 할 수 있는 방안이 유연하다.

실제 서비스에 적용할 때 고려해야 할 사항 정리
Index(Database) Type(Table) Document(Row)

  • 어느 범위를 Index에 저장할 것인가?
  • Type 의 범위?

기본적으로 ElasticSearch는 Index를 엮거나 Type 들을 엮어서 검색 가능함

Index, Type 분류 방법 2가지

Multiple Index Single Type

  • 인덱스 내에서 검색하는 것이 효율적이고 빠르다. 각 인덱스가 다른 인덱스에 분산되어 있기 때문에 각 샤드에서 데이터 크기가 작아야한다.
  • Multiple Index 데이터 조합 검색은 오버헤드 존재 한다.
    – 인덱스를 통해 더 많은 샤드에 쿼리를 보내 컴파일하고 사용자에게 다시 보내야 하기 때문이다.
  • 데이터의 양이 적다면 사용하지 않아야한다.
    – 양이 적은데 그것을 위해서 추가적인 샤드, 더 많은 스토리지가 필요하기 때문이다.
  • 만약 데이터셋이 크고 쿼리한번 하는데 오래 걸린다면 추천
    – 독점 샤드숫자가 많기 때문에 특정 데이터들을 저장할 수 있다. 때문에 ES가 쿼리를 처리할 때 편해진다.

Single Index Multiple Type

  • 1개의 인덱스 안에 있는 5개의 샤드(디폴트)에 더 많은 데이터가 저장된다.
    – 이것은 다른 type 을 조합하는 쿼리를 날릴때 오버헤드가 적다는 것을 의미한다.
    – 하지만 샤드 크기가 상당히 커질 거다.
  • 샤드가 커지면 필터를 많이 사용해야 하기 때문에 검색 시간이 길어 질 수 있다.
  • 만약 데이터 용량이 1TB 이상이 된다면 사용하면 안된다.
  • 다른 인덱스 또는 여러 샤드에 데이터를 분산하지 않는다면 사용 시 비효율을 가져온다.
  • 저장하는 데이터가 작다면 이 방법을 사용한다. 추가적인 샤드나 스토리지가 필요하지 않기 때문이다.

적용할 방식

다음과 같은 이유로 Single Index Multiple Type 방식으로 사용한다.

  1. 여러 인덱스를 사용하기에는 데이터 양이 많지 않다.
  2. 5분에 한번 들어오는 데이터를 많은 양의 Dataset으로 나누지 않고 그대로 사용할 것이기 때문이다.
  3. type을 Device로 나누는게 좋을 것 같다.

Device 별로 Aggregation을 구하는 일이 더 많을 것이기 때문에

  1. 3번과 같은 방법으로 Dataset을 구성한다면 굳이 Label을 사용할 필요가 없어 진다.
  2. 매장당 Device가 1개라고 가정하면 Type 안에 Label을 사용하지 않아도 Type만으로도 충분히 구분 가능 하다.

라즈베리파이 + 미세먼지 & 온습도센서

라즈베리파이 3b 의 핀 맵은 아래와 같습니다. 라즈베리파이 다른 버전을 사용하게된다면 아래 핀맵과 비교하여 같은 Pin 에다가 연결 하시면 됩니다.

아래 이미지는 PM5003의 PinMap 입니다.
사용하는 핀은 1,2,4,5 번 핀을 사용할 예정입니다. 나머지 핀은 사용하지 않기 때문에 선을 연결 할 경우 모든 선을 연결 하지 말고 4개의 선만 연결하여 사용 할 수 있도록 개량이 필요합니다.

DHT11 은 따로 핀맵을 작성하지 않겠습니다. 제가 가지고 있는 DHT11의 경우 센서에 GND, VCC, DATA 핀이 어디인지 표시되어 있고, 또한 DHT11 센서의 경우 매우 많은 제조사가 있기 때문에 자신이 사용하는 제조사의 Pin Map을 참고 하시기 바랍니다. 대부분 3개에서 4개의 핀이 존재하는데 저는 위의 3개의 핀만 사용합니다.

라즈베리 파이와 센서 연결

라즈베리 파이와 센서 연결은 위의 보이는 핀에다가 연결해 주시면 됩니다.
PMS5003 센서의 1,2 번 PIN은 VCC, GND이기 때문에 라즈베리 파이의 VCC, GND 원하시는 곳에 연결 해 주시면 됩니다.
DHT11 센서의 VCC, GND 도 마찬가지로 원하는 곳에 연결해 주면 됩니다.
라즈베리 파이의 VCC, GND 어디에 연결해도 상관 없습니다.

라즈베리파이 SETTING

소프트 웨어는 Raspbian Stretch 를 사용하였습니다. 하드웨어는 라즈베리파이 3B 버전을 사용하였습니다.
라즈비안 Stretch 버전은 라즈베리파이 홈페이지에 가셔서 다운로드 받으시면 됩니다.

위에서 받은 라즈비안 이미지를 SD카드에 업로드 시켜야 합니다.
업로드 할때 사용한 프로그램은 Etcher 라는 프로그램입니다.

SD카드는 라즈비안 이미지 크기보다 큰 용량을 가지고 있어야 합니다.

정상적으로 이미지를 SD카드에 옮기고 라즈베리를 켜게 되면 위와 같은 바탕화면이 나오게 됩니다.

라즈베리 파이 3 시리얼/블루투스 문제

라즈베리 파이 3로 오면서 무선랜/블루투스 기능 추가를 위해 Broadcom의 BCM43438 칩을 사용하고 있는데 무선랜
기능은 SDIO로 연결하지만 블루투스 기능은 UART0으로 연결을 하고 시리얼 포트 핀은 UART1에 연결되도록 구조가
변경되었다. 그러므로 그 동안 아무 문제 없이 시리얼 포트를 사용하던 프로그램이 라즈베리 파이 3로 오면서 블루투스를 사용하게
되면 제대로 동작을 할 수 없게 되어 버린 것이다.

위와 같은 상황이 벌어져 Setting을 약간 해야 된다고 합니다.
라즈베리 파이 3 시리얼/블루투스 문제
자세한 내용은 위의 블로그를 참조하면 좋습니다.

‘/boot/config.txt’에 ‘dtoverlay=pi3-disable-bt’

를 추가

UART1을 사용하지 않는걸로 해서 블루투스를 비활성화 시킨다.

"sudo systemctl disable hciuart""sudosystemctldisablehciuart"

명령을 추가로 실행합니다.

RaspberryPi Wifi Setting

Sudo iwlist wlan0 scan

랜카드가 인식한 와이파이 리스트가 리스트업 됩니다. 검색된 무선 공유기의 ESSID를 확인한 후 연결할 와이파이를 확실하게 적어놔야 합니다.

sudo nano /etc/network/interfaces

SSID와 비밀번호를 확실하게 입력해줘야 합니다. 그렇지 않으면 연결이 되지 않기 때문입니다.
또한 inet 옆에 DHCP를 입력하면 Address나 netmask, gateway를 따로 입력하지 않아도 됩니다.
static을 입력했다면 망에서 IP충돌이 나지 않는 번호를 입력하여 직접 할당을 해줘야한다. 유선이 아니더라도 무선에서도 가능하다.

여기서 wpa-ssid 옆에 문자열로 방금 전 검색한 ESSID를 입력하고 wap-psk 옵션에는 비밀번호를 입력한다.

노드 JS 설치

인터넷 연결이 잘 되었다면 Node.js 를 설치 합니다.
라즈베리파이 콘솔에서 다운로드 명령어를 입력합니다.

pi@raspberrypi:~ $ wget
https://nodejs.org/dist/v8.11.3/node-v8.11.3-linux-armv7l.tar.xz

위의 다운로드한 파일을 아래와 같은 명령어를 통해 압축을 풉니다.

pi@raspberrypi:~ $ tar -xvf node-v8.11.3-linux-armv7l.tar.xz

압축을 풀면 같은 이름의 디렉토리가 생성됩니다. 원하는 곳으로 옮기고자 한다면 아래의 명령어를 치면 됩니다.

pi@raspberrypi:~ $ sudo mv node-v8.11.3-linux-armv7l /{원하는 폴더}

기존에 설치되어 있던 nodejs를 삭제합니다.

pi@raspberrypi:~ $ sudo apt remove nodejs

정상적으로 동작하는지 확인합니다.

pi@raspberrypi:~ $ node --version

v8.11.3
